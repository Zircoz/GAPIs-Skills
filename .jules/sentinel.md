## 2024-05-24 - Unrestricted Webhook Testing Utilities (SSRF & DoS)
**Vulnerability:** A CLI utility script (`test_webhook.py`) accepted any user-supplied URL and executed a `requests.post()` call to it without domain validation or a timeout. This creates Server-Side Request Forgery (SSRF) and Denial of Service (DoS) risks, especially if the utility is used in automated contexts.
**Learning:** Even simple developer utility scripts can introduce critical vulnerabilities like SSRF if they handle external URLs dynamically. Relying on "developer discretion" is insufficient; the tool itself must enforce secure boundaries.
**Prevention:** Always validate external URLs against a strict allowlist (e.g., `startswith("https://chat.googleapis.com/")`) and enforce explicit timeouts (e.g., `timeout=10`) for all outbound HTTP requests in utilities and CLIs.

## 2024-05-24 - Webhook Token Leakage via HTTP Client Exceptions
**Vulnerability:** A script using the `requests` library caught `requests.exceptions.RequestException` and printed the raw exception object (`e`) to the console. When `requests` encounters connection errors, the exception string includes the full URL, inadvertently exposing the sensitive `key` and `token` query parameters of Google Chat webhook URLs in error logs.
**Learning:** HTTP client libraries (like `requests`, `axios`, etc.) often embed full request details (including sensitive URL parameters) into their exception messages. Logging or printing these exceptions raw breaks the "fail securely" principle and leads to credential leakage.
**Prevention:** Never log or print raw HTTP client exception objects when the URL contains sensitive query parameters. Catch the exception and print a static, sanitized error message (e.g., `print("Network error occurred")`) without exposing the exception's internal details.
